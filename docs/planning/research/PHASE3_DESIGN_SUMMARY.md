## 🎯 **핵심 발견사항**

### **실제 상황: 반품이 매우 빈번함**
```
분석 대상: Coupang거래명세표.pdf
  구매 상품: 4개
  반품 상품: 1개 (칫솔 18,500원)
  결과: 최종 결제액 38,460원 (원래 41,410원)

→ 반품 처리가 **필수 기능**, 선택 기능이 아님!
```

### **기존 설계의 부족점**
```
원본 설계: 반품/취소를 고려하지 않음
  - 결제액과 거래명세 금액만 맞추려고 함
  - 반품 상품을 어떻게 처리할지 미정

개선: 반품/취소 전략 3가지 제시
  A. 수동 선택 (사용자가 매번 체크)
  B. 자동 감지 (쿠팡 API) - 현실성 낮음
  C. 보정 거래 ⭐ 추천 (취소 영수증으로 자동 감지)
```

## 🔧 **Phase 3 최종 설계 (반품 처리 포함)**

### **주요 특징**

| 항목 | 설명 |
|------|------|
| **입력** | 쿠팡 거래명세표 + 취소 영수증 (선택) |
| **처리** | 상품별 분해 + 반품 상품 환급 처리 |
| **출력** | N개 거래 (수령 상품 + 반품 환급) |
| **Step** | 7단계 (기존 5 → 추가 2 Step) |
| **특수** | 반품 환급은 Refund 타입으로 처리 |

### **Step-by-Step**

```
Step 1: 입력 (파일 업로드)
  └─ 쿠팡 거래명세표 (PDF/이메일/CSV)

Step 2: 파싱 (상품 추출)
  └─ 자동으로 상품 리스트 추출
  └─ 금액 검증

Step 3: 반품/취소 확인 ⭐ NEW
  ├─ "반품하신 상품이 있으신가요?"
  ├─ 없음 → 다음으로
  └─ 있음 → 취소 영수증 업로드 (자동 파싱)

Step 4: 수령/반품 상태 확인 ⭐ NEW
  ├─ 각 상품마다 [수령] or [반품(환급)] 선택
  ├─ 반품 상품 자동 감지 + 수동 확인
  └─ 최종 금액 검증

Step 5: 카테고리 분류
  └─ 수령 상품만 카테고리 선택 (반품은 자동)

Step 6: 미리보기
  ├─ 수령 상품: 일반 표시
  └─ 반품 환급: 빨간색, "-" 표시

Step 7: 완료
  └─ 거래 생성 (원본 숨김, N개 항목 생성)
```

### **결과 예시**

```
입력: "쿠팡" 38,460원 (2026-02-20)

거래명세표:
  ├─ 오넛티 땅콩버터 11,900원 ✓
  ├─ 모나리자 키친타올 4,710원 ✓
  ├─ 루치펠로 칫솔 18,500원 ✗ (반품)
  └─ 크리넥스 마스크 6,300원 ✓

↓ 분해 후

가계부 기록:
  ├─ "오넛티 땅콩버터" 11,900원 [생활비]
  ├─ "모나리자 키친타올" 4,710원 [생활비]
  ├─ 🔴 "[반품환급] 루치펠로 칫솔" -18,500원 [기타/환급]
  └─ "크리넥스 마스크" 6,300원 [생활비]

합계: 38,460원 ✓ (카드와 일치)
```

## 📊 **반품 처리 메커니즘: 3가지 전략**

### **전략 C 추천: 보정 거래 (Adjustment Transaction)**

```
원리:
  1. 거래명세표의 모든 상품 임시 분해
  2. 취소 영수증 파싱 (반품 상품 식별)
  3. 반품 상품을 음수 거래로 생성 (Refund 타입)
  4. 최종 금액 자동 검증

장점:
  ✅ 자동화 수준 높음
  ✅ 사용자 개입 최소 (취소 영수증만 필요)
  ✅ 정확도 높음 (금액으로 역산)

단점:
  ⚠️ 취소 영수증 파싱 필요
  ⚠️ 여러 반품 있으면 식별 복잡

구현:
  ├─ Step 3에서 취소 영수증 업로드
  ├─ OCR/정규식으로 반품 상품 추출
  ├─ 상품명 퍼지 매칭 (80% 이상)
  ├─ 수동 선택 UI (매칭 실패 시)
  └─ 자동으로 Refund 타입 할당
```

## 🛠️ **기술적 구현 사항**

### **DB 스키마 변경**

```sql
-- transactions 테이블 확장
ALTER TABLE transactions
ADD COLUMN type VARCHAR(20),           -- 'expense', 'refund', 등
ADD COLUMN is_refund BOOLEAN,          -- 반품/환급 여부
ADD COLUMN refund_reason VARCHAR(200), -- 반품 사유
ADD COLUMN refund_date TIMESTAMP;      -- 반품일

-- transaction_breakdowns 테이블 확장
ALTER TABLE transaction_breakdowns
ADD COLUMN refund_amount NUMERIC,  -- 환급 금액
ADD COLUMN refund_reason VARCHAR;  -- 환급 사유
```

### **새로운 API 엔드포인트**

```typescript
// 취소 영수증 파싱
POST /api/transactions/:id/breakdown/parse-cancellation
  ├─ 파일 업로드 (PDF/이메일)
  └─ 반품 항목 자동 추출

// 반품 상품 매칭
POST /api/transactions/:id/breakdown/match-refunds
  ├─ 거래명세 상품 + 취소 영수증 반품 항목 매칭
  └─ 퍼지 매칭 + 수동 선택 지원
```

### **UI 변경**

```
기존: 5-Step 다이얼로그
  1. 입력
  2. 파싱
  3. 분류
  4. 미리보기
  5. 완료

개선: 7-Step 다이얼로그
  1. 입력
  2. 파싱
  3. 반품/취소 확인 ⭐ NEW
  4. 수령/반품 상태 ⭐ NEW
  5. 분류
  6. 미리보기 (반품 시각화)
  7. 완료
```

## 💡 **핵심 인사이트**

### **1. 반품은 예외가 아닌 일상**
```
쿠팡에서 여러 상품을 사면 일부 반품할 확률이 높음
→ 반품 처리가 "꼭 있어야 하는" 기능
```

### **2. 취소 영수증이 연결고리**
```
쿠팡 앱: 반품 상태 확인 (앱 로그인 필요)
이메일: 자동으로 "반품 완료" 알림 (모든 사용자)

→ 이메일 취소 영수증 파싱이 가장 현실적
```

### **3. 금액으로 역산하는 것이 가장 정확**
```
사용자가 일일이 "어떤 상품을 반품했나?" 선택하는 것보다
금액 기반으로 자동 계산 → 수동 확인이 더 정확

최종 검증: 수령 상품 + 반품 환급 = 카드 결제액
```

### **4. UI는 복잡하지만 필수**
```
5-Step → 7-Step (2개 추가)
- Step 3: 반품/취소 확인
- Step 4: 수령/반품 상태

사용자가 한번 정확히 입력하면
이후 분석/통계가 모두 정확해짐
```
