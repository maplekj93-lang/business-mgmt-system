# 🔄 쿠팡 반품/취소 처리 전략 (Coupang Return & Cancellation Strategy)

> **작성일:** 2026-03-11
> **목적:** 쿠팡 분해기의 반품/취소 상황 처리 방안 정립
> **배경:** 실제 쿠팡 거래명세표에서 반품이 빈번하게 발생 → 처리 방법 필요

---

## 🎯 **현황 분석**

### **문제 상황**
```
시나리오: 사용자가 여러 상품을 한꺼번에 구매했는데, 일부만 반품

카드사 결제:  "쿠팡" 38,460원 (원래는 41,410원, 할인 2,950원)
  ├─ 오넛티 땅콩버터: 11,900원 ✓ (배송 받음)
  ├─ 모나리자 키친타올: 4,710원 ✓ (배송 받음)
  ├─ 루치펠로 칫솔: 18,500원 ✗ (반품 함)
  └─ 크리넥스 마스크: 6,300원 ✓ (배송 받음)

문제점:
  1. 반품한 상품(칫솔 18,500원)은 실제로 지출이 아님
  2. 하지만 카드사에는 결제로 기록됨
  3. 환불 처리까지 포함하면 거래가 복잡함
  4. 취소 영수증은 한꺼번에만 나옴 → 개별 항목과 매칭 어려움
```

---

## 📊 **반품/취소 유형 분류**

### **Type 1: 부분 반품 (Partial Return)**
```
구매 시: 상품 4개 (41,410원)
배송 후: 1개 반품 (18,500원)
최종 결제: 22,910원

처리:
  원본 카드: "쿠팡" 38,460원 (이미 할인 적용됨)

  분해 시:
  ├─ 오넛티: 11,900원 [생활비] (수령)
  ├─ 모나리자: 4,710원 [생활비] (수령)
  ├─ 루치펠로: -18,500원 [반품환급] (반품) ← Refund
  └─ 크리넥스: 6,300원 [생활비] (수령)

  합계: 38,460원 ✓
```

### **Type 2: 전체 주문 취소 (Full Cancellation)**
```
구매 시: "쿠팡" 50,000원 결제
배송 전: 전체 취소 요청
결과: 결제 자동 취소 (가계부에 기록되지 않음)

처리:
  카드사 내역: 나타나지 않음 (자동 취소)
  또는 나타나더라도 Refund로 처리
```

### **Type 3: 일부 취소 + 배송 (Partial Order Cancel)**
```
구매 시: 3개 상품 (30,000원)
배송 전: 1개 취소
배송: 2개만 배송

카드사 결제: "쿠팡" 20,000원 (이미 취소분 제외)

처리:
  간단함 - 그냥 2개 상품만 분해
```

## 🔧 **해결 방안: 3가지 전략**

### **전략 A: 수동 선택 (Manual Selection)**

**사용자가 직접 선택:**
```
UI: 거래명세표 파싱 후 상품 목록 표시

사용자 선택:
  ☑️ 오넛티 땅콩버터 11,900원 [수령]
  ☑️ 모나리자 키친타올 4,710원 [수령]
  ☐ 루치펠로 칫솔 18,500원 [반품]  ← 체크 해제 = 환급 처리
  ☑️ 크리넥스 마스크 6,300원 [수령]

결과:
  ├─ 수령 상품 3개: 11,900 + 4,710 + 6,300 = 22,910원
  ├─ 반품 상품 1개: -18,500원 (환급, Refund 타입)
  └─ 합계: 38,460원 ✓ (카드 결제액과 일치)
```

**장점:**
- 가장 정확함 (사용자 확인)
- UI 간단

**단점:**
- 사용자가 매번 선택해야 함
- 실수 가능성

---

### **전략 B: 자동 감지 (Auto Detection)**

**쿠팡 앱/이메일에서 반품 상태 자동 추출:**

```
쿠팡 API 또는 이메일 분석:
  - 쿠팡 앱에서 "반품 요청됨" 또는 "반품 완료" 상태 확인
  - 이메일: "반품 안내", "환불 처리됨" 등의 키워드 감지

자동 태깅:
  ├─ 오넛티: [배송 완료] ✓
  ├─ 모나리자: [배송 완료] ✓
  ├─ 루치펠로: [반품 요청] → 자동으로 "Refund" 타입으로 처리
  └─ 크리넥스: [배송 완료] ✓
```

**장점:**
- 자동화 (사용자 개입 최소)
- 정확도 높음 (공식 상태에서 추출)

**단점:**
- 쿠팡 API 필요 (현실성 낮음)
- 이메일 파싱 복잡도 높음
- 초기 구현 어려움

---

### **전략 C: 보정 거래 (Adjustment Transaction) ⭐ 추천**

**카드 결제액 기반 역산:**

```
원리:
  카드사 최종 결제: 38,460원 (반품 환급 이미 적용됨)
  거래명세표 총액: 41,410원
  차이: 2,950원 (할인) + 18,500원 (반품환급) = 21,450원

처리:
  Step 1: 거래명세표의 모든 상품을 일단 분해
    ├─ 오넛티: 11,900원
    ├─ 모나리자: 4,710원
    ├─ 루치펠로: 18,500원
    └─ 크리넥스: 6,300원
    합계: 41,410원

  Step 2: 카드 결제액과 비교
    카드: 38,460원
    거래명세: 41,410원
    차이: -2,950원 (할인)

  Step 3: 취소/반품 내역서 업로드
    "루치펠로 칫솔 18,500원 반품"

  Step 4: 자동으로 보정
    - 루치펠로 항목의 타입을 "Refund"로 변경
    - 또는 음수 거래로 생성 (-18,500원)

  최종 결과:
    ├─ 오넛티: 11,900원 [생활비]
    ├─ 모나리자: 4,710원 [생활비]
    ├─ 루치펠로: -18,500원 [환급]  ← Refund 타입
    └─ 크리넥스: 6,300원 [생활비]
    합계: 38,460원 ✓
```

**장점:**
- 비교적 간단한 구현
- 사용자는 취소 영수증만 업로드
- 자동 계산으로 정확도 높음

**단점:**
- 취소 영수증 파싱 필요
- 여러 반품이 있으면 식별 필요

```
│                                       │

```

```
  ├─ "모나리자 키친타올" 4,710원 [생활비]
```

```sql

```

## 🔗 **거래명세표 ↔ 취소 영수증 매칭**

### **문제: 상품명이 미세하게 다를 수 있음**
```
거래명세표: "루치펠로 코랄리프 양면 미세모 칫솔, 12개입, 1개"
취소 영수증: "루치펠로 코랄리프 양면 칫솔"

→ 자동 매칭 어려움
```

### **해결: 퍼지 매칭 (Fuzzy Matching)**
```typescript
import { levenshteinDistance } from 'library';

// 유사도 계산:
const similarity = 1 - (levenshteinDistance(a, b) / Math.max(a.length, b.length));

// 80% 이상 유사하면 매칭
if (similarity >= 0.8) {
  match(product, cancellation);
}
```

### **수동 선택 UI (Fallback)**
```
자동 매칭 실패 시:

┌──────────────────────────────────┐
│ 반품 항목 매칭                    │
├──────────────────────────────────┤
│                                  │
│ 취소 영수증:                      │
│ "루치펠로 코랄리프 양면 칫솔"     │
```

## 💡 **추가 고려사항**

### **부분 환급 (Partial Refund)**
```
상품 가격: 30,000원 (수량 2개)
부분 반품: 1개만 반품 → 15,000원 환급

처리:
  거래명세: "상품 A" 30,000원 (수량 2)
  반품: 1개 → -15,000원

  분해 결과:
  ├─ "상품 A (1개)" 15,000원 [수령]
  └─ "상품 A (1개)" -15,000원 [반품환급]
```

### **재배송/교환 (Re-shipping/Exchange)**
```
케이스: 반품 후 새 상품으로 교환

처리:
  Step 1: 원본 상품 반품 처리
    "상품 A" -20,000원 [반품]

  Step 2: 새 상품 도착
    다시 쿠팡 분해기로 처리
    "상품 B (교환)" 20,000원 [구매]
```
