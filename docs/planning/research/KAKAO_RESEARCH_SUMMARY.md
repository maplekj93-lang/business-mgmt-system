## 🔍 **핵심 발견사항**

### 1. **쿠팡과 카카오페이는 "정반대 문제"**

| 구분 | 쿠팡 | 카카오페이 |
|------|------|----------|
| **문제의 성격** | 집약 (aggregation) | 분산 (distribution) |
| **해결 방향** | 분해 (breakdown) | 통합 (mapping) |
| **도구명** | 쿠팡 분해기 | 카카오페이 매퍼 |
| **자동화 수준** | 중간 (수동 지원) | 높음 (자동 매칭) |
| **우선순위** | 2순위 (Phase 3) | 1순위 (Phase 2) |

### 2. **Double Count Prevention이 핵심**

```
원본 설계의 부족점:
  "충전 시점에는 지출이 없다"

상세화 후:
  - 은행 충전: Transfer (이체)
  - 카카오페이 결제: Expense (지출)
  - 개인 통계에서 제외할 항목: excluded_from_personal = true
```

### 3. **두 도구의 선택적 사용**

```
시나리오 1: 카카오페이만 사용
  → 카카오페이 매퍼만 필요

시나리오 2: 쿠팡 직접 구매
  → 쿠팡 분해기만 필요 (선택사항)

시나리오 3: 카카오페이로 쿠팡 구매
  → 카카오페이 매퍼 (필수)
     + 쿠팡 분해기 (선택, 필요시)
```

## 🎯 **가능성 평가 (Feasibility Assessment)**

### **카카오페이 매퍼: ✅ 높은 가능성 (High Feasibility)**

✅ **강점**
- 거래내역서는 구조화된 데이터 (CSV)
- 자동 매칭 로직이 명확함 (거래일 + 금액)
- Smart Tagging V2와 명확한 연계점
- 오류 처리도 정의 가능 (불일치 감지)

⚠️ **주의사항**
- 거래일 오차 문제 (카드 승인일 vs 실제 결제일)
- 환급/수수료 처리 (거래 유형이 많음)
- 통화 문제 (환전 거래의 경우)

### **쿠팡 분해기: ⚠️ 중간 가능성 (Medium Feasibility)**

✅ **강점**
- 상품명 기반 카테고리 제안 가능
- Smart Tagging과 자연스러운 연계
- 사용자가 수동 입력 옵션 있음

⚠️ **주의사항**
- 상세정보 입력이 필수 (자동화 한계)
- 파서 개발 필요 (이메일, CSV, 스크린샷 등)
- Double Count 방지가 복잡함
