# 카카오페이 매칭 기능 트러블슈팅 및 수정 로그

**문서 정보**
- **작성일**: 2026년 3월 11일
- **관련 기능**: 카카오페이 가맹점 거래내역 매퍼 (Phase 2)
- **주요 이슈**: `카카오페이머니_거래내역확인서.xlsx` (41건) 업로드 시, 기존 은행/카드 내역과 부분적으로만 매칭(초기 9건)되는 현상 진단 및 해결 과정.

---

## 타임라인 및 수정 과정 (로그)

### 1단계: 매칭 키워드 확장 (단순 일치 -> 포함 검색)
- **현상**: 매칭 엔진(`matcher.ts`)이 `description?.includes('카카오페이')` 조건으로만 원본 거래를 찾고 있었음.
- **분석**: DB `transactions` 테이블을 `ILIKE '%카카오%'`로 조회한 결과, '카카오페이' 외에도 '주식회사 카카오', '카카오T', '카카오스타일', '카카오(선물하기)' 등 다양한 적요가 사용됨을 확인.
- **조치 (수정 1)**:
  - `matcher.ts`의 `isKakaoName` 검사 로직을 확장하여 '카카오' 텍스트가 포함된 모든 내역을 매칭 대상으로 삼음.
  - 타행 이체 오탐지를 막기 위해 '뱅크'(예: 카카오뱅크) 키워드가 포함된 내역은 예외 처리함.
- **결과**: 엔진 레벨에서 매칭 후보군 점유가 넓어졌으나, 여전히 실제 UI에서는 9건만 매칭됨.

### 2단계: 데이터 Fetch 로직 (쿼리) 수정 누락 발견
- **현상**: 엔진(Matcher) 조건은 완화되었으나 매칭 건수의 변화가 없음.
- **분석**: UI 컴포넌트(`ledger-import-widget.tsx`)에서 DB로부터 원본 내역을 처음 가져올 때(Fetch), 쿼리 조건이 여전히 `.ilike("description", "%카카오페이%")`로 고정되어 있었음. 즉, 엔진에는 애초에 '카카오페이'라고 적힌 데이터만 전달되고 있었음.
- **조치 (수정 2)**:
  - `ledger-import-widget.tsx`의 Supabase DB 조회 구문을 `.ilike("description", "%카카오%")` 및 `.not("description", "ilike", "%뱅크%")`로 수정하여 데이터 공급 범위를 엔진 조건과 일치시킴.
- **결과**: 매칭 후보군이 정상적으로 41건 전부 엔진으로 전달됨. 그러나 여전히 최종 매칭 건수가 비정상적으로 적음. 매칭(비교) 조건 자체에 논리적 결함이 있음을 발견.

### 3단계: Date(날짜) 비교 로직 및 Timezone(표준시) 오차 해결 (Core Issue)
- **현상**: 매칭 조건 중 `diffDays <= 1` (원본 거래일과 카카오페이 내역일의 차이가 하루 이하) 로직이 대다수 거래를 걸러내고(Reject) 있었음.
- **분석**:
  1. 카카오페이 CSV 파서가 날짜를 `2026.02.14` 형식으로 추출.
  2. `new Date('2026.02.14')` 파싱 시 V8엔진/브라우저에 따라 'Invalid Date' 처리되거나 표준시(UTC) 변환 과정에서 오프셋 9시간(KST)이 빠지면서 전날 오후(예: 2026-02-13T15:00:00Z)로 인식됨.
  3. DB의 표준 Date 형식(`2026-02-14T00:00:00.000Z`)과 시간차(`diffTime`)를 계산할 때, 이 타임존 오프셋 차이로 인해 24시간 + 1밀리초 오차가 발생함.
  4. 이를 `Math.ceil(오차 / 하루)`로 올림 처리하는 기존 로직 탓에, 동일한 날짜임에도 "2일 차이"로 계산되어 `diffDays <= 1` 조건을 통과하지 못함.
- **조치 (수정 3)**:
  - **문자열 치환**: `matcher.ts` 내에서 카카오 내역 날짜 문자열의 `.`을 `-`로 필수 치환하여 `Invalid Date` 방지(`formattedRowDate = row.date.replace(/\./g, '-')`).
  - **UTC 자정 강제 초기화**: 타임존 오차와 시/분/초 차이를 무시하기 위해, DB Date와 Row Date 양쪽 모두 `Date.UTC(연, 월, 일)`을 사용하여 **완벽한 UTC 자정 타임스탬프**를 새로 생성.
  - **부동소수점 수학 계산 수정**: `Math.ceil` 연산자를 `Math.round`로 바꾸어 초단위 미세 오차가 일수를 넘기는 현상을 원천 방지함 (`diffDays = Math.round(diffTime / 1Day)`).
- **최종 결과**: 날짜 형식 불일치와 타임존 오프셋으로 인한 오탐지가 잡히며 논리 결함 완전 해소. (이후 다시 CSV 업로드 시 정확히 확장된 데이터 풀 내에서 Date-Amount 복합 일치를 시도하게 됨)

## 교훈 및 개선 방향 (Lessons Learned)
- **One-Shot 원칙 (맥락 통합)**: 데이터베이스 조회 계층(Widget/Query)과 비즈니스 로직 계층(Matcher Engine) 간 필터 조건이 이중 파편화 되어있어 발생한 실수임. 필터 조건 분리 시 캡슐화를 통해 엔진 내부로 은닉하거나 쿼리 변수로 통일해야 함.
- **날짜(Date) 객체 취급 주의**: 외부 CSV/Excel 업로드 데이터의 String Date 형식을 다룰 때는 `new Date()`에 바로 넘기지 말고, ISO 8601 표준 포맷(`YYYY-MM-DD`) 정규화 및 UTC 변환 혹은 Date-FNS 등의 전문 라이브러리를 활용한 방어적 프로그래밍이 필수적임.
