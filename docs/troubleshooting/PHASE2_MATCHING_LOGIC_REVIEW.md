## 🎯 **현재 상황 정리**

### **9건이 매칭된 이유**

```
카드사 결제: "카카오페이" 50,000원 (은행 계좌에서 출금)
  ↓ [매칭]
카카오 엑셀: 여러 상품 지출 합계 50,000원 (같은 날짜)

→ 1:1 매칭 성공

이 케이스는:
  - 카드로 직접 결제 (계좌에서 출금)
  - 거래 당일 결제 완료
  - 금액이 정확히 일치

조건을 모두 만족하는 거래만 매칭됨
```

### **32건이 매칭되지 않은 이유**

```
카카오 엑셀: "커피" 3,000원, "우유" 2,000원, ... (여러 건)

하지만 은행 DB에서 이 거래들과 매칭할
원본 카드 결제 기록이 존재하지 않음

→ 논리적으로 매칭 불가능 (시스템 정상 작동)
```

## 🔍 **안티그래비티의 2가지 제시안 검토**

### **제시안 1: 매칭 안 된 32건도 새 지출로 추가 (추천)**

#### 장점
```
✅ 카카오페이 거래의 100% 가계부 반영
✅ 사용자 입장에서 보면 "왜 내 지출이 안 보여?"라는 불만 해결
✅ 가계부 완전성 향상
```

#### 단점 (그리고 해결책)
```
❌ Double Count Prevention 필수
   예: 은행에 "카카오페이 50,000원 충전" 있고
       카카오페이에 "50,000원 지출" 있으면
       → 총 100,000원으로 계산 (이중 지출!)

해결책:
  - 카카오페이 충전액은 "Transfer" 타입으로 분류
  - 직접 지출은 "Expense" 타입으로 분류
  - 개인 순자산 통계에서 Transfer는 제외
  - excluded_from_personal = false인 Expense만 계산
```

#### 구현 복잡도
```
중간 정도
  - 32건의 새 거래 객체 생성
  - 적절한 카테고리 할당 (Smart Tagging 연계)
  - Double Count Prevention 로직 보강
```

**평가:** ✅ **추천할 만함**

```typescript
// 기존: 매칭되는 거래만 저장
const matchedTransactions = results;  // 9건

// 추가: 매칭 안 된 거래도 처리
const unmatchedRows = kakaoPayRows.filter(row =>
  !matchedTransactions.some(m => m.kakaoPayRow === row)
);  // 32건

// 새로운 거래로 생성
const newTransactions = unmatchedRows.map(row => ({
  date: row.date,
  amount: row.amount,
  description: `[카카오페이] ${row.merchant}`,
  type: 'expense',
  category_id: smartTagging(row.merchant),  // Smart Tagging
  source_raw_data: { ...row },
  excluded_from_personal: false
}));

await uploadNewTransactions(newTransactions);
```
