# 가계부 임포트 & 거래 자동분류 딥리서치 프롬프트

---

## 배경 컨텍스트 (읽고 시작할 것)

우리는 부부 공동 가계부 + 프리랜서 ERP 앱을 개발 중입니다.
현재 엑셀(카드사/은행 명세서)을 임포트해서 거래내역을 관리하는 기능이 있고,
미분류 거래를 카테고리로 분류하는 수신함 기능이 있습니다.

**현재 분류 방식:**
- 가맹점명(raw_name) + owner_type + 수입/지출 타입 기준으로 그룹핑
- 3단계 룰 엔진: 정확히 일치(Exact) → 키워드 포함(Contains) → 과거 이력(History)
- 그룹 단위로 카테고리 일괄 지정

**핵심 문제:**
- 쿠팡, 카카오페이, 배달의민족 같은 가맹점은 같은 이름이어도 매번 구매 내용이 달라서 하나의 카테고리로 묶을 수 없음
- 카카오페이/네이버페이 결제는 실제 가맹점명이 파싱 안 됨
- 임포트 시점에 owner_type(누구 거래인지) 자동 판별이 안 됨

각 섹션마다 **구체적인 근거**와 **실제 동작 방식**을 포함해야 합니다. 추측 금지.

### 섹션 1. 타 앱 벤치마크 — 자동분류 방식 비교

아래 앱들이 거래 자동분류를 **실제로 어떻게** 구현하는지 조사하세요.

조사 대상:
- **국내**: 뱅크샐러드, 토스(가계부), 카카오뱅크 가계부, 네이버 가계부
- **해외**: 머니포워드(일본), YNAB, Copilot(미국), Monarch Money

각 앱마다 아래 항목을 조사:
1. 분류 방식 — 규칙 기반인지, ML인지, 사용자 학습인지, 혼합인지
2. 그룹핑 전략 — 같은 가맹점 여러 건을 어떻게 처리하는지
3. **카카오페이/네이버페이 같은 간편결제** — 실제 가맹점 정보를 어떻게 처리하는지
4. **배달의민족/쿠팡** — 카테고리 불일정 케이스를 어떻게 처리하는지
5. 오분류 수정 UX — 틀렸을 때 어떻게 고치고, 학습에 반영되는지
6. 초기 설정 — 처음 쓸 때 분류 규칙을 어떻게 세팅하는지

### 섹션 2. 국내 카드사/은행 원본 데이터 품질 조사

국내 카드사/은행이 제공하는 거래 명세 데이터의 **실제 가맹점명 상세도**를 조사하세요.

조사 항목:
1. **카드사별 가맹점명 상세도 차이**
   - 삼성카드, 현대카드, BC카드, 신한카드, 국민카드, 롯데카드
   - 예: "스타벅스"만 오는지 "스타벅스 강남점"까지 오는지
   - 백화점의 경우 "롯데백화점"만 오는지 "롯데백화점 식품관 B2"까지 오는지

2. **간편결제 데이터 한계**
   - 카카오페이로 결제 시 카드 명세서에 실제 가맹점명이 나오는지
   - 네이버페이, 토스페이, 삼성페이 동일 조사
   - 간편결제사가 API로 실제 가맹점 정보를 제공하는지 여부

3. **카드 명세서 vs 은행 이체 내역 데이터 품질 차이**
   - 카드 명세서: 가맹점명 품질
   - 통장 이체 내역: 적요(description) 품질
   - 어느 쪽이 분류하기 더 좋은 데이터인지

4. **오픈뱅킹/마이데이터 API 활용 시 데이터 품질**
   - 스크래핑 방식과 마이데이터 API 방식의 데이터 품질 차이
   - 마이데이터에서 MCC 코드(가맹점 업종 코드)를 제공하는지

### 섹션 3. 거래 자동분류 기법 조사

거래 자동분류에 사용되는 기법들을 조사하세요.

조사 항목:
1. **가맹점명 → 카테고리 매핑 방식**
   - 단순 키워드 매칭 vs 퍼지 매칭(오타 허용) vs 임베딩 기반 유사도
   - 공개된 가맹점-카테고리 데이터셋이 있는지 (예: 금융결제원 MCC 코드 활용)
   - LLM을 거래 분류에 활용하는 사례 (예: GPT API로 가맹점명 → 카테고리)

2. **금액/패턴 기반 보조 분류**
   - 금액 구간으로 카테고리 추정하는 방식 (예: 편의점 1만원 미만 → 식비)
   - 시간대/요일 패턴 활용 (예: 점심 시간대 → 식비)
   - 정기결제 자동 감지 방법 (동일 금액 반복 패턴)

3. **간편결제 가맹점 파싱**
   - 카카오페이 영수증 API나 웹훅이 실제 가맹점 정보를 제공하는지
   - 네이버페이 가맹점 정보 접근 방법
   - 이를 활용한 가계부 앱 사례

4. **사용자 학습 기반 분류**
   - Federated Learning 적용 사례
   - 개인 이력만 쓰는 방식 vs 익명화된 공통 모델 방식
   - 초기 cold start 문제 해결법

### 섹션 4. 현재 구조 발전 방향 제안

위 조사를 바탕으로, 아래 현재 시스템의 구체적인 개선 방향을 제안하세요.

현재 시스템 스펙:
- Next.js 14 (App Router) + Supabase (PostgreSQL)
- 엑셀 임포트 방식 (마이데이터 미사용)
- 3단계 룰 엔진: Exact Match → Contains Match → History Match
- 그룹핑 기준: GROUP BY (raw_name, owner_type, type)
- 사용자: 부부 2명 (소규모, 개인 이력 기반이면 충분)

제안 요청 항목:
1. **그룹핑 전략 개선** — 묶어도 되는 것과 건별 처리해야 하는 것을 어떻게 자동 판별할지
2. **룰 엔진 → 경량 ML 전환 가능성** — Supabase + Edge Function 환경에서 가능한지
3. **LLM 활용** — 가맹점명을 LLM API에 넣어서 카테고리 추론하는 방식의 실용성 (비용, 속도, 정확도)
4. **임포트 파이프라인 개선** — 임포트 시점에 자동으로 더 많이 처리할 수 있는 것들
5. **간편결제 대응** — 카카오페이/네이버페이 실제 가맹점 파싱이 기술적으로 가능한지
